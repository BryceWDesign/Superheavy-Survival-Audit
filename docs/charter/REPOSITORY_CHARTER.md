# Repository Charter

## Repository name

**Superheavy-Survival-Audit**

## Why this repository exists

This repository exists to make superheavy-nuclei discussions harder to bluff.

It is designed to collect evaluated and curated inputs, preserve provenance,
carry uncertainty forward where possible, and generate audit artifacts that
help an outside reader distinguish:

- what is measured
- what is literature-derived
- what is transformed
- what is repository-defined
- what remains uncertain
- what fails under stress-testing

The repository is not built to maximize spectacle. It is built to reduce the
distance between claim and evidence.

## Primary objectives

1. Preserve source traceability from ingestion to output.
2. Compare neighboring superheavy chains under a shared data model.
3. Quantify observability and decay-legibility under explicit assumptions.
4. Represent route-feasibility at a non-operational abstraction level.
5. Expose model sensitivity, disagreement, and instability.
6. Produce reproducible benchmark and release artifacts.

## Product philosophy

The product of this repository is not a single score.

The product is a set of disciplined artifacts:

- normalized datasets
- provenance records
- benchmark packs
- sensitivity outputs
- uncertainty-aware plots
- route bottleneck abstractions
- release notes that explain what changed and why

## What success looks like

The repository is succeeding when it becomes easier for a technically literate
reader to answer questions such as:

- what is directly supported by evaluated data?
- where do model interpretations diverge?
- which rankings are robust and which are fragile?
- where is ambiguity driven by branching, uncertainty, or missing data?
- which candidate paths appear most informative to measure next?
- what assumptions dominate the outcome?

## What failure looks like

The repository is failing when it:

- sounds stronger than it is
- buries provenance
- hides uncertainty
- mixes evidence and invention without labels
- treats patents or speculative mechanisms as proof
- rewards dramatic narratives over conservative interpretation

## Near-term mission

The first serious version of the repository should establish:

- a validated schema layer
- evaluated-data ingestion
- provenance contracts
- observability and ambiguity scoring
- route-feasibility abstraction
- uncertainty-aware modeling
- literature-facing benchmarks
- reproducible outputs

## Long-term mission

The long-term mission is not to prove exotic claims.

The long-term mission is to become a reliable audit and prioritization
instrument for superheavy-candidate interpretation, measurement value, and
model disagreement under explicit constraints.

## Operating discipline

Every module should make at least one of these things better:

- traceability
- interpretability
- benchmarkability
- falsifiability
- uncertainty handling
- release reproducibility

If it does not, it does not belong here.
