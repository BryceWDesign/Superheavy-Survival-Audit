# Research BOM

## Purpose

This document is the repository's research bill of materials.

It is not a claim of completeness over the entire nuclear-data ecosystem.
It is a structured disclosure of the main ingredients this repository is built
to use, compare, or reference so that outside reviewers can quickly see what the
project depends on conceptually and operationally.

This file is separate from the software SBOM because research inputs are not
the same thing as software dependencies.

## 1. Repository identity

- Repository: `Superheavy-Survival-Audit`
- Repository version target: `0.1.0a0`
- License: `Apache-2.0`
- Author / maintainer: `Bryce Lovell`

## 2. Core upstream data families

### 2.1 Evaluated and curated nuclear data
Primary intended families:

- IAEA LiveChart of Nuclides
- AMDC AME2020
- AMDC NUBASE2020
- ENSDF and structured ENSDF-derived exports

Why they matter:

- canonical nuclide identity
- evaluated half-lives
- decay relationships
- branch structure
- Q-values
- mass-excess values
- neighboring-chain comparison context

Repository role:

- ingest
- normalize
- preserve provenance
- compare
- benchmark

Repository non-claim:

- upstream evaluated values are not generated by this repository
- repository transformations do not improve upstream facts by magic
- repository outputs remain bounded by upstream uncertainty and coverage

## 4. Research logic families

### 4.1 Observability logic
Current internal observability layers include:

- decay-chain observability score
- branch competition ambiguity score
- daughter-chain legibility profile

Purpose:

- help judge interpretability of a candidate chain
- expose ambiguity instead of hiding it
- keep chain quality and chain readability explicit

Boundary:

- these are repository-defined surrogates
- they are not direct measurements
- they do not guarantee detection

### 4.2 Survival-audit logic
Current internal survival layers include:

- baseline survival-audit score
- Bayesian coefficient layer
- Monte Carlo sensitivity summaries
- leave-one-term-out ablation summaries
- posterior predictive checks

Purpose:

- keep the score structure explicit
- expose assumption dependence
- make scoring layers easier to falsify

Boundary:

- these are not stability proofs
- these are not production cross sections
- these are not laboratory viability claims

### 4.3 Mass-residual comparison logic
Current internal mass-residual layer includes:

- reference-model prediction comparison
- residual clustering summary
- residual spread and sign agreement summary
- consensus score

Purpose:

- compare observed AME-style values against reference predictions
- track agreement or disagreement structure
- avoid pretending one residual is the whole story

Boundary:

- this is not a new mass model
- this is not proof of shell closure
- this is not proof of stabilization

### 4.4 Route logic
Current internal route layers include:

- route-feasibility scoring
- structured constraint assessment
- information-gain ranking

Purpose:

- prioritize what might be informative to compare next
- keep bottlenecks explicit
- avoid opaque one-number route rhetoric

Boundary:

- not a laboratory recipe
- not an execution protocol
- not an experimental operations guide

## 8. Failure modes the BOM is meant to guard against

This BOM exists partly to reduce the following failure modes:

- hidden source dependence
- unclear provenance
- silent scope creep
- unbounded claim inflation
- pretending surrogate layers are measured quantities
- route romanticism without bottleneck disclosure
- benchmark rhetoric without benchmark structure

## 9. Exclusions

This repository does not claim to include:

- all possible nuclear-data sources
- all public superheavy models
- all reference mass models
- all public GitHub tooling
- all patent families
- all experimental constraints

This BOM is a disciplined disclosure file, not a false claim of total coverage.
